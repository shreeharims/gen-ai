## **1. Explain the need of Security Policies in detail**

### **Answer:**

A **Security Policy** is a set of rules and guidelines that define how an organization protects its data, systems, and network.

### **Need for Security Policies:**

1. **Protection of Data**

   * Ensures confidentiality, integrity, and availability of data.
   * Prevents unauthorized access and data leaks.

2. **Defines Responsibilities**

   * Clearly assigns roles to employees (admin, user, etc.).
   * Avoids confusion during security incidents.

3. **Prevents Security Attacks**

   * Provides guidelines for password usage, access control, etc.
   * Reduces risks of hacking, malware, and insider threats.

4. **Legal Compliance**

   * Helps organizations follow laws and regulations.
   * Avoids legal penalties.

5. **Incident Response**

   * Defines steps to handle security breaches.
   * Helps in quick recovery.

6. **User Awareness**

   * Educates employees about safe practices.
   * Reduces human errors (biggest weakness btw).

### **Conclusion:**

Security policies are essential to maintain a secure and controlled environment in any organization.

## **2. Compare Intrusion Detection System (IDS) and Intrusion Prevention System (IPS)**

### **Answer:**

| Feature           | IDS                         | IPS                        |
| ----------------- | --------------------------- | -------------------------- |
| **Definition**    | Detects attacks             | Detects + prevents attacks |
| **Action**        | Alerts only                 | Blocks attack immediately  |
| **Placement**     | Monitors traffic            | Placed inline in network   |
| **Response Time** | Slow (manual action needed) | Fast (automatic response)  |
| **Example**       | Logs suspicious activity    | Drops malicious packets    |

### **Key Points:**

* IDS = Alarm system (just shouts “problem!”)
* IPS = Security guard (actually stops it)

### **Conclusion:**

IPS is more advanced than IDS because it provides real-time protection.

## **3. Classify and explain different types of Firewalls**

### **Answer:**

Firewalls are security systems that monitor and control network traffic.

### **Types of Firewalls:**

1. **Packet Filtering Firewall**

   * Filters packets based on IP, port, protocol.
   * Fast but less secure.

2. **Stateful Inspection Firewall**

   * Tracks active connections.
   * More secure than packet filtering.

3. **Proxy Firewall (Application Level)**

   * Acts as intermediary between user and server.
   * Hides internal network.

4. **Next-Generation Firewall (NGFW)**

   * Advanced firewall with features like IDS/IPS, deep inspection.
   * Provides strong security.

5. **Hardware Firewall**

   * Physical device.
   * Used in large networks.

6. **Software Firewall**

   * Installed on individual systems.
   * Protects single device.

### **Conclusion:**

Firewalls act as the first line of defense in network security.

## **4. Examine Buffer Overflow and Countermeasures**

### **Answer:**

### **Buffer Overflow:**

A buffer overflow occurs when a program writes more data than a buffer can hold, overwriting adjacent memory.

### **Causes:**

* Lack of bounds checking
* Unsafe functions (e.g., `gets()`, `strcpy()`)

### **Effects:**

* Program crash
* Execution of malicious code
* System compromise

### **Countermeasures:**

1. **Bounds Checking**

   * Validate input size before storing.

2. **Safe Functions**

   * Use `fgets()`, `strncpy()` instead of unsafe ones.

3. **Stack Canaries**

   * Detect memory corruption.

4. **ASLR (Address Space Layout Randomization)**

   * Randomizes memory locations.

5. **Non-executable Stack**

   * Prevents execution of injected code.

### **Conclusion:**

Proper coding practices and memory protection techniques can prevent buffer overflow attacks.

## **8. Compare DoS attack and DDoS attack**

### **Answer:**

| Feature            | DoS (Denial of Service)    | DDoS (Distributed DoS)                  |
| ------------------ | -------------------------- | --------------------------------------- |
| **Source**         | Single system              | Multiple systems (botnet)               |
| **Traffic Volume** | Low                        | Very high                               |
| **Detection**      | Easier                     | Difficult                               |
| **Attack Power**   | Limited                    | Extremely powerful                      |
| **Example**        | One system flooding server | Thousands of infected devices attacking |

### **Key Points:**

* DoS = One idiot attacking
* DDoS = Army of idiots attacking together

### **Conclusion:**

DDoS is more dangerous due to large-scale coordinated attacks.

## **10. Analyze Attack Surfaces of Set-UID Programs**

### **Answer:**

### **Set-UID Programs:**

Set-UID programs run with the **privileges of the file owner (usually root)**.

### **Attack Surfaces:**

1. **Environment Variables**

   * Attackers manipulate variables like `PATH`, `LD_LIBRARY_PATH`.
   * Can execute malicious programs.

2. **Input Validation Issues**

   * Improper input checking leads to injection attacks.

3. **Buffer Overflow**

   * Overflow can allow privilege escalation.

4. **File Access Vulnerabilities**

   * Improper file permissions can expose sensitive files.

5. **Race Conditions**

   * Time gap between check and use exploited.

### **Conclusion:**

Set-UID programs are highly sensitive and must be carefully designed to avoid privilege escalation attacks.

## **15. Analyse Cross-Site Scripting (XSS) attack and countermeasures**

### **Answer:**

### **Cross-Site Scripting (XSS):**

XSS is a web attack where an attacker injects **malicious scripts (usually JavaScript)** into a trusted website.

### **Types of XSS:**

1. **Stored XSS**

   * Script is stored in server (e.g., database).
   * Executes when users load the page.

2. **Reflected XSS**

   * Script comes from user input (URL/search).
   * Immediately reflected in response.

3. **DOM-Based XSS**

   * Occurs in client-side scripts (browser).

### **Effects:**

* Cookie theft
* Session hijacking
* Redirect to malicious sites

### **Countermeasures:**

1. **Input Validation**

   * Filter user input.

2. **Output Encoding**

   * Encode HTML/JavaScript characters.

3. **Use Secure Frameworks**

   * Avoid direct DOM manipulation.

4. **Content Security Policy (CSP)**

   * Restricts script execution.

5. **HttpOnly Cookies**

   * Prevent access via JavaScript.

### **Conclusion:**

XSS attacks exploit user trust, so proper input/output handling is critical.

## **29. Risks and Vulnerabilities with Environment Variables and Set-UID Programs**

### **Answer:**

### **Risks:**

1. **PATH Manipulation**

   * Attacker changes path to execute malicious program.

2. **LD_LIBRARY_PATH Exploit**

   * Loads malicious libraries.

3. **IFS Variable Attack**

   * Alters command parsing behavior.

4. **Privilege Escalation**

   * Gains root access via Set-UID programs.

5. **Unsafe System Calls**

   * Executes unintended commands.

### **Vulnerabilities:**

* Lack of input validation
* Blind trust in environment variables
* Poor coding practices

### **Countermeasures:**

1. **Clear Environment Variables**
2. **Use Absolute Paths**
3. **Drop Privileges when not needed**
4. **Validate Inputs**
5. **Avoid unsafe functions**

### **Conclusion:**

Environment variables can be dangerous if not handled securely in privileged programs.


## **33. List and explain the types of Buffer Overflow attack**

### **Answer:**

### **Types of Buffer Overflow Attacks:**

1. **Stack-Based Buffer Overflow**

   * Occurs in stack memory.
   * Overwrites return address.
   * Most common type.

2. **Heap-Based Buffer Overflow**

   * Occurs in heap memory.
   * Targets dynamically allocated memory.
   * Harder to exploit but dangerous.

3. **Integer Overflow**

   * Occurs when arithmetic exceeds limit.
   * Leads to incorrect buffer size allocation.

4. **Format String Attack**

   * Improper use of format functions like `printf()`.
   * Allows reading/writing memory.

5. **Off-by-One Overflow**

   * Writes one byte beyond buffer.
   * Seems small but can be exploited.

### **Conclusion:**

Different overflow types target different memory areas, but all can lead to serious system compromise.

## **34. Cross-Site Request Forgery (CSRF) Attack**

### **Answer:**

### **CSRF Attack:**

CSRF tricks a user into performing unwanted actions on a web application **without their consent**.

### **How it Works:**

* User logs into a website.
* Attacker sends malicious link/request.
* Browser automatically sends stored cookies.
* Action is performed as if user requested it.

### **Example:**

* Clicking a link that transfers money from your account without your knowledge.

### **Effects:**

* Unauthorized transactions
* Data modification
* Account misuse

### **Countermeasures:**

1. **CSRF Tokens**

   * Unique token for each request.

2. **SameSite Cookies**

   * Restricts cookie sharing.

3. **User Authentication Checks**

   * Re-verify sensitive actions.

4. **CAPTCHA**

   * Prevents automated attacks.

5. **Check HTTP Referer/Header**

   * Validate request source.

### **Conclusion:**

CSRF exploits user trust, so proper request validation is essential.
